# QAレビュー規格セット

第三者検証の品質基準をAIレビューに注入する。判定は常にevidence-only（根拠を提示できない指摘は出力しない）。

## ISO/IEC 25010 スコアリング（5カテゴリ運用）
UX/プロダクトレビューでは以下5特性で0〜100スコア＋根拠を出す:
1. **使用性**（操作の学習性・エラー防止・アクセシビリティ/WCAG 2.1）
2. **性能効率性**（応答時間・リソース・容量）
3. **信頼性**（障害許容・回復性・データ保全）
4. **セキュリティ**（機密性・インテグリティ・否認防止）
5. **保守性**（モジュール性・解析性・修正容易性）

出力形式: カテゴリ別スコアカード＋指摘リスト＋改善提案。総合点は単純平均にせず、プロダクト特性に応じた重み付けを明示する。

## ISTQB severity 分類
全指摘に必ず付与:
- **Critical**: データ損失・セキュリティ侵害・主要機能停止
- **High**: 主要ユースケースの阻害、回避策が困難
- **Medium**: 回避策あり、限定的影響
- **Low**: 軽微なUI不整合・改善提案レベル

severityとpriorityを混同しない。priorityは別軸（ビジネス判断）として扱う。

## Whittaker ツアーモデル（探索的テスト観点）
テスト観点生成時は以下から該当ツアーを選んで適用:
- Guidebook Tour（仕様書通りの操作）/ Money Tour（売りの機能）
- Landmark Tour（主要機能間の遷移）/ Intellectual Tour（複雑な入力）
- FedEx Tour（データのライフサイクル追跡）/ Saboteur Tour(破壊的操作)
- Back Alley Tour（使われない機能）/ All-Nighter Tour（長時間稼働）

## AIレビュープロンプトへの組み込み手順
1. システムプロンプトに上記3規格の判定基準を明記
2. 「JSONのみで出力、根拠（evidence）フィールド必須」を強制
3. evidenceが空の指摘はパース後に除外
4. イテレーション比較（delta）がある場合は前回スナップショットとの差分を別セクションで出す

## 報告フォーマット
指摘1件 = `{id, category(25010), severity(ISTQB), evidence, 再現手順, 改善案}`。Jira CSV / GitHub Issuesへのエクスポートを想定したフラット構造にする。
