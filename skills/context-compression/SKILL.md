# コンテキスト圧縮（出力規約とスクリプト優先原則）

トークンは「推論」に使い、「作業ログ・生データの垂れ流し」に使わない。以下の3原則を常に適用する。

## 原則1: 長大な出力は3層で要約する

調査・監査・集計など出力が30行を超えそうな場合は、以下の3層構造にする。

| 層 | 内容 | 分量 |
|---|---|---|
| 結論 | 質問への直接の答え・判定 | 1-3行 |
| 根拠 | 結論を支える主要な事実（ファイル名:行番号つき） | 5-10行 |
| 詳細 | 全件データ・生ログ・網羅表 | **会話に出さずファイルへ書き出す** |

- 詳細層は `docs/` または作業ディレクトリに Markdown で書き出し、会話にはパスと1行サマリだけを出す
- 「全件見せて」と明示されない限り、会話へ詳細層を展開しない

## 原則2: 全読みより grep/glob で絞る

- 「まず全ファイルを読んで把握」を禁止。**検索 → 該当ファイルのみ読む** の順を守る
- 読む前に必ず自問する: 「この情報は grep 1回で取れないか？」
- ディレクトリ構造の把握は `find`/glob の一覧で済ませ、中身は必要になったファイルだけ開く
- 大きいファイルは行範囲指定で部分読みする（該当箇所を grep で特定してから）

```bash
# 悪い例: skills/ 配下を1ファイルずつ全部読む
# 良い例: まず横断検索で当たりをつける
grep -rn "ECC" skills/ --include="SKILL.md" -l
```

## 原則3: 決定論的作業は必ずスクリプト化する

集計・検索・変換・カウント・整形など「同じ入力なら同じ出力になる作業」を、モデルの手作業（目視で数える・1件ずつ書き写す）で行わない。

- **集計**: 行数カウント・ファイル数・出現頻度 → `wc` / `grep -c` / `awk`
- **検索**: 横断的なパターン抽出 → `grep -rn` / glob
- **変換**: 一括リネーム・形式変換・表の生成 → シェル/Python スクリプトを書いて実行
- 3回以上繰り返す操作は、その場で使い捨てスクリプトにしてから回す
- スクリプトの出力も原則1に従う（全件はファイルへ、会話にはサマリ）

## 判断フロー

```
出力が30行を超える？ ── yes → 3層要約（詳細はファイルへ）
ファイルを読もうとしている？ ── yes → 先に grep/glob で絞ったか確認
同じ操作を3回以上繰り返す？ ── yes → スクリプト化
```

## 出力

このスキル適用時は、(1) 結論→根拠の要約、(2) 詳細を書き出したファイルパス、(3) 使用したスクリプト（再実行可能な形）を提示する。
