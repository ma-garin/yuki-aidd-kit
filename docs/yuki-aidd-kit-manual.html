AIDD Kitを迷わず使うためのガイド

このHTMLは、AI駆動開発に慣れていない新卒1年目の人でも、何を見て、何を実行し、どこで止まればよいかが分かるように書いた手順書です。

AIDD Claude Code Codex ECC QA Playwright

はじめに

AIDD Kitは、毎回ゼロから「どう進めるんだっけ」と悩まないための開発キットです。 AIに作業を頼む時のルール、品質チェック、よく使う開発パターン、ECCの使い分けをまとめています。

まず覚えること

このキットは「AIに全部丸投げする道具」ではありません。 仕様を決める、実装する、テストする、振り返る、という流れを崩さないための道具です。

使う時は、難しく考えずに次の一文から始めてください。 

このプロジェクトに合うECCだけ選んで

すると、`ecc-daily-router` が対象プロジェクトを見て、毎回使うものと必要な時だけ使うものを分けます。

AIDD Kitは何をする道具か

一言でいうと、利用者の開発スタイルに合わせた「AI開発の作業台」です。 とくに次のようなプロジェクトで力を出します。

よくある開発 キットが助けること
QA document generatorのようなQA文書生成ツール Python、Playwright、pytest、E2E、仕様ドリフト確認
UX audit toolのようなUX監査ツール 画面チェック、アクセシビリティ、ISO 25010、ISTQB severity
React finance PWAのような個人PWA React、Vite、localStorage、GitHub Pages、スマホ対応
QA portalのような業務ポータル Django、Pythonテスト、セキュリティ、業務導線レビュー
QA repository research toolのような調査・評価ツール OSS調査、評価基準、LLM eval、レポート化
なぜECCを全部入れないのか

ECCは大きな道具箱です。全部読むと便利そうに見えますが、作業ごとの読み込み量が増えます。 そのため、このキットでは「毎回使うDAILY」と「必要時だけ見るLIBRARY」に分けます。

よくある誤解: 配置しただけで見た目が自動的に変わる？

いいえ、なりません。キットを配置（グローバル導入 or プロジェクト配布）しても、 それだけで既存の画面の色やレイアウトが勝手に書き換わることはありません。

「配置する」ことの意味は、AIがいつでも参照できる状態にすることです。 実際にデザインが反映されるのは、次のような流れになります。

  1. キットを配置する（`install.sh` または `export-project.sh`）
  2. 普段通りAIに「画面を作って」「デザインを整えて」のように依頼する
  3. AIが `skills/design-system/SKILL.md` を自動的に参照する（「デザインどうする」「UIを作る」等の言葉に反応して自動で発火するよう設計されています。毎回「design-systemスキルを使って」と言う必要はありません）
  4. その結果、色・余白・角丸・モーダルの作り方などがキット内の値・パターンに沿ったものになる

つまり「配置＝自動適用」ではなく「配置＝AIが正しい答えを迷わず出せるようになる」という意味です。 作業そのもの（画面を作る・直す）はこれまで通りAIに依頼する必要があります。

最初にやること

初回は、キットが正しく置かれているか確認します。 まだ中身を理解していなくても大丈夫です。まずはファイルの場所を確認します。

1. キットの場所へ移動する

cd <YOUR_WORKSPACE>/yuki-aidd-kit

2. 中身を見る

ls

`README.md`、`INDEX.md`、`skills/`、`docs/`、`scripts/` が見えれば基本はOKです。

3. 配置確認をする

./scripts/verify.sh

これは「必要なファイルがあるか」を見るための確認です。 もし失敗したら、エラー文をそのままAIに渡して「原因だけ教えて」と聞きます。

4. hooksの動作を確認する

./scripts/test-hooks.sh

保存時の自動チェック（hooks）が正しく動くかを8ケースでテストします。 「PASS=8 / FAIL=0」なら合格です。

注意

`npm ci`、ブラウザE2E、Playwrightのインストール、GitHub Pagesの配信確認は重くなりがちです。 必要な時だけ、理由を確認してから実行します。

導入方式（2種類）

このキットには2つの導入方式があります。どちらか一方を選ぶのではなく、目的に応じて両方使います。

方式 向いている場面 コマンド
① グローバル導入 自分のPC1台で、複数の個人プロジェクトを日常的に横断する `./scripts/install.sh`（`~/.claude/` に配置）
② プロジェクト配布 Codex、リモート/使い捨てのClaude Code環境、チームメイトへ渡す等、`~/.claude`が効かない・望ましくない場面 `./scripts/export-project.sh <対象プロジェクト>`
なぜ2つ必要か

グローバル導入は「自分のPCさえ整えれば、どのプロジェクトを開いても効く」形です。 一方でCodexは基本的にプロジェクト直下の`AGENTS.md`を読みますし、 使い捨てのクラウド環境やチームメイトのPCには自分の`~/.claude`は存在しません。 そのため、配布したい対象プロジェクトには②で `.claude/` と `AGENTS.md` / `CLAUDE.md` を 直接書き出し、そのプロジェクトのgitにコミットして持ち運びます。

プロジェクト配布の手順

./scripts/export-project.sh /path/to/target-project
cd /path/to/target-project
git add .claude AGENTS.md CLAUDE.md
git commit -m "chore: add AIDD Kit"

これで、対象プロジェクトをCodexで開いても、リモートのClaude Codeで開いても、 チームメイトがcloneしても、install不要でそのままキットが効きます。

注意

プロジェクト配布は「書き出した時点のコピー」です。キット本体を後から更新しても、 配布済みのプロジェクトには自動反映されません。更新を反映したい時は `export-project.sh` をもう一度実行します（既存ファイルは`.bak`に退避されます）。

毎日の使い方

普段の開発は、次の流れで進めます。

  1. 対象プロジェクトの `AGENTS.md` または `README.md` を読む
  2. `ecc-daily-router` で、そのプロジェクトに合うECC資産を選ぶ
  3. 大きな変更なら `sdd-ecc-workflow` で仕様とタスクに分ける
  4. 実装する
  5. `test-automation` で動作確認する
  6. `done-gate` で完了判定する
  7. 学びがあれば `retro` で残す

AIに依頼する時は、次のように言うと安定します。 

このプロジェクトのAGENTS.mdとREADME.mdだけ読んで、
今回の変更に必要なECC DAILYを選んでください。
まだ実装はしないで、方針だけ出してください。

これで、いきなり大きな調査や実装に入る事故を減らせます。

スキルの選び方

`skills/` は、AIが作業する時の専門メモです。 人間でいうと「この仕事ではこの手順で進める」というチェックリストに近いです。

やりたいこと 使うスキル 一言でいうと
どのECCを使うか選びたい `ecc-daily-router` 道具箱から必要な道具だけ選ぶ
仕様から作りたい `sdd-ecc-workflow` spec、plan、tasksに分ける
UIや品質をレビューしたい `qa-review-standards` ISO 25010とISTQBで見る
画面のデザインを整えたい `design-system` 色・余白・モーダル・ダーク対応まで統一された見た目にする
テストを書きたい `test-automation` Playwrightやpytestで確認する
完成と言ってよいか確認したい `done-gate` 完了前の最終チェック
AI出力を評価したい `agent-eval` LLMの品質を測る
作業の学びを残したい `retro` 次回のために改善点を残す
トークンを節約して進めたい `context-compression` 要約は3層、探索より検索、集計はスクリプト

スキル全体の一覧は、リポジトリの `INDEX.md` にタグと参照コスト付きで載っています。

初心者向けの考え方

スキル名を全部覚える必要はありません。 「仕様」「レビュー」「テスト」「完成」「振り返り」の5語だけ覚えておけば、だいたい進められます。

スラッシュコマンド一覧

`claude-code/commands/` には、明示的に呼び出すための短い入口があります。 たとえば `/ecc-daily` と入力すると、今のプロジェクトに必要なECC資産を選ぶ流れになります。

コマンド 使うタイミング
`/ecc-daily` 今のプロジェクトに合うECC資産を選びたい時
`/app-scan` `<APP_WORKSPACE>` 全体の棚卸しを軽く更新したい時
`/sdd-start` 新機能を仕様から作りたい時
`/new-pwa` 個人用PWAを新しく作る時
`/qa-review` UI、仕様、テスト観点をレビューしたい時
`/doc-search` 技術ドキュメントを調べたい時
`/eval` AI出力やRAGの品質を評価したい時
`/retro` 作業の振り返りを残したい時
`/token-check` 読み込み過ぎやトークン消費を見直したい時
`/compact-work` 長い調査・集計をトークン節約モードで進めたい時

ECCとの関係

ECCは、たくさんのスキル、ルール、エージェントを持つ大きな開発支援セットです。 AIDD Kitは、その中から利用者のプロジェクトでよく使うものを選びやすくするための薄い案内役です。

正確な対応表の場所

プロジェクト別にどのECCを使うかの正式な対応表は `docs/ECC-ASSET-MAP.md` にあります。 このページのリストは雰囲気をつかむための例なので、実際の分類はそちらを見てください。

DAILYにするECC（例）

  • `skills/e2e-testing`: PlaywrightやE2E確認
  • `skills/browser-qa`: ブラウザでの表示・操作確認
  • `skills/accessibility`: alt、label、コントラストなど
  • `skills/frontend-patterns`: HTML、CSS、React、画面実装
  • `skills/react-patterns`: React系アプリ
  • `skills/vite-patterns`: ViteとGitHub Pages
  • `skills/python-patterns`: Python実装
  • `skills/python-testing`: pytest、ruff、mypy
  • `skills/django-patterns`: Djangoポータル
  • `skills/security-review`: 秘密情報、XSS、入力検証
  • `skills/verification-loop`: 完了前の検証
  • `skills/deep-research`: 技術調査やOSS比較
  • `skills/eval-harness`: AI出力の評価

LIBRARYでよいECC

Swift、Go、Java、Rust、PHP、ブロックチェーン系などは、今の主要プロジェクトでは毎回読まなくて大丈夫です。 必要になった時だけ呼びます。 

このプロジェクトはReact/Viteです。
ECCはDAILYだけ使って、SwiftやDjango系はLIBRARY扱いにしてください。

プロジェクト別の使い方

QA document generator

Python、Flask、Playwright、pytestが中心です。 画面仕様書やテスト入力文書を作るので、E2EとQAレビューが重要です。

  • Kit: `test-automation`, `qa-review-standards`, `done-gate`
  • ECC: `python-patterns`, `python-testing`, `e2e-testing`, `browser-qa`

UX audit tool

PlaywrightでWebページを取得し、UI/UXを監査します。 アクセシビリティとブラウザ確認を強めに使います。

  • Kit: `qa-review-standards`, `design-system`, `test-automation`
  • ECC: `browser-qa`, `accessibility`, `frontend-patterns`, `e2e-testing`

React finance PWA / investment hypothesis PWA

React、Vite、localStorage、GitHub Pagesの系統です。 データ保存、スマホUI、export/importを必ず確認します。

  • Kit: `personal-pwa`, `nfr-standards`, `done-gate`
  • ECC: `react-patterns`, `vite-patterns`, `frontend-patterns`, `security-review`

QA portal

Django、HTMX、SQLiteの業務ポータルです。 権限、入力、データ境界、業務用語の分かりやすさを確認します。

  • Kit: `streamlit-rag-app`, `qa-review-standards`, `agent-eval`
  • ECC: `django-patterns`, `python-testing`, `backend-patterns`, `security-review`

QA repository research tool

QA会社向けにOSSリポジトリを発見・分類・評価します。 調査の根拠、評価基準、レポートの説明責任が大事です。

  • Kit: `agent-eval`, `code-doc-search`, `sdd-ecc-workflow`
  • ECC: `deep-research`, `eval-harness`, `security-review`, `verification-loop`

品質チェックの考え方

このキットでは、品質チェックを3つに分けます。 混ぜると判断がぼやけるので、別々に見ます。

見るもの 使うもの 質問
コードが動くか `test-automation` テストは通るか
画面が使いやすいか `qa-review-standards` 利用者が迷わないか
AI出力が正しいか `agent-eval` 根拠に基づいているか
完成と言ってよいか `done-gate` 未確認の項目は残っていないか
一番大事なルール

「手元で見た感じ大丈夫」は完了ではありません。 テスト、E2E、スクリーンショット、レビュー、完了ゲートのどれで確認したかを残します。

  • ロジックを変えたら、単体テストまたはpytest/vitestを確認する
  • UIを変えたら、ブラウザまたはスクリーンショットで確認する
  • AI出力を変えたら、evalまたはゴールデンデータで確認する
  • APIキーや個人情報が絡む時は、security-reviewを通す
  • 最後にdone-gateで「未完了項目なし」を確認する

困った時の進め方

何から読めばいいか分からない

AGENTS.mdとREADME.mdだけ読んで、
このプロジェクトの入口を3行で教えてください。

AIが広く読みすぎる

node_modules、venv、dist、output、test-resultsは読まないでください。
変更対象に近いファイルだけ確認してください。

どのECCを使えばいいか分からない

/ecc-daily
このプロジェクトに必要なECC資産をDAILYとLIBRARYに分類してください。

完成したか不安

done-gateで完了判定してください。
未確認項目があれば、未完了として列挙してください。

次回に同じ失敗をしたくない

/retro
今回のKeep / Problem / Tryを整理し、lessons.mdに残す内容を出してください。

調査結果が長すぎて読みにくい

/compact-work
結論と根拠だけ会話に出して、詳細はファイルに書き出してください。

キット自体を直したい・育てたい

キット本体の開発は `docs/Roadmap.md` が作業台帳です。 未完了のチェックボックスに、対象ファイル・完了条件・検証手順が書いてあるので、その通りに進めます。

用語集

用語 意味
AIDD AI Driven Development。AIを使って、仕様、実装、テスト、レビューを回す開発の進め方。
ECC AI開発向けの大きな道具箱。skills、rules、agents、hooksなどを持つ。
DAILY そのプロジェクトで毎回使うもの。最初から読ませてよい。
LIBRARY 必要になった時だけ読むもの。削除ではなく、参照用として残す。
Skill AIに「この作業はこう進める」と教える手順書。
done-gate 完成と言ってよいかを確認する最終チェック。
eval AIの出力品質を測ること。コードのテストとは別物。
E2E End to End。ユーザー操作に近い形で、画面の流れ全体を確認するテスト。
