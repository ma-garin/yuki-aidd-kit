# Vision — yuki-aidd-kit

## 一文で

**AI 駆動開発（AIDD）の「進め方」自体を資産化し、どの AI エージェントに渡しても同じ品質・同じトークン効率で開発が回る状態を作る。**

## このキットは何か

個人の開発活動（QA ツール・個人 PWA・単一 HTML ツール・Streamlit 業務アプリ）で繰り返し使う開発規約・品質基準・作業手順を、AI エージェントが直接読めるスキル／コマンド／hook の形で保存したものである。コードのテンプレート集ではなく、**「AI にどう作業させるか」の運用知識**が本体。

## 解決したい問題

| 問題 | キットの答え |
|---|---|
| セッションごとに規約・品質基準を説明し直している | スキルとして永続化し、発火条件（description）で自動適用 |
| AI が探索的にファイルを読みトークンを浪費する | context-compression / トークン規律 / DAILY・LIBRARY の2層参照 |
| 「動いた」＝完成と誤判定される | done-gate / test-automation / agent-eval の3段判定 |
| レビュー品質が AI の気分で変わる | ISO/IEC 25010・ISTQB・evidence-only を規格として注入 |
| 外部資産（ECC）が膨大で全部読むと破綻する | ecc-daily-router + ECC-ASSET-MAP でプロジェクト別に絞り込む |
| 改善知見がセッション終了で消える | retro スキルで lessons.md に蓄積 → キットへ還流 |

## 目指す状態（3つの到達点）

1. **エージェント可搬性** — Claude Code でも他のエージェント（Codex 等）でも、同じファイルを読ませれば同じ振る舞いになる。ツール固有名に依存しない記述を保つ
2. **ゼロコンテキスト継続性** — 前提知識のないモデルが `INDEX.md` → `docs/Roadmap.md` の順に読むだけで、キット自体の開発を続行できる
3. **自己改善ループ** — retro → lessons.md → スキル/コマンド/hook への反映、が月次で回り、キットが使うほど強くなる

## やらないこと（Non-Goals)

- **不特定多数への OSS 公開・汎用フレームワーク化**（ドキュメントサイト運営、外部コントリビュータ受け入れ、スター獲得目的の一般公開はしない）。一方で、**自分のプロジェクト・チームメイトへのプロジェクト単位の配布は行う**（`scripts/export-project.sh`。詳細は「配置の2層」を参照）。プレースホルダ `<APP_WORKSPACE>` 等は導入者が置換する
- ECC 資産の同梱・複製（ECC は外部キットとして参照のみ。真実源は `docs/ECC-ASSET-MAP.md`）
- 実プロジェクトのコード資産の保管（キットは進め方のみ。成果物は各プロジェクトリポジトリ）

## 配置の2層（グローバル運用 と プロジェクト配布）

このキットは2つの配置場所を使い分ける。両立が前提であり、どちらか一方に統一しない。

| 層 | 置き場所 | 導入 | 効く範囲 | 目的 |
|---|---|---|---|---|
| **グローバル層** | `~/.claude/`・`~/.codex/` | `scripts/install.sh` | 自分のPC1台・複数プロジェクト横断 | 普段使いの一貫性。キット更新が即座に全プロジェクトへ反映される |
| **プロジェクト配布層** | 対象プロジェクト直下の `.claude/`・`AGENTS.md`・`CLAUDE.md` | `scripts/export-project.sh <target>` | Codex／リモート・エフェメラルな Claude Code 環境／CI／teammate の clone 先 | install 不要でその場で効く。対象プロジェクトの git にコミットして持ち運ぶ |

プロジェクト配布層は書き出した時点のスナップショットであり、キット本体の更新には自動追従しない（再エクスポートで同期する）。これは配布の性質上避けられないトレードオフとして許容する。

## 価値の判定基準

キットへの変更は「**新しいセッションの立ち上がりが速くなるか／トークンあたりの成果が増えるか／品質判定が再現可能になるか**」のいずれかに寄与しない限り採用しない。
